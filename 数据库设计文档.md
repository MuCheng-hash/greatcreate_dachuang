## 3. MySQL 表结构说明

## 3.1 administrative_region 行政区表

用途：

- 存储行政区划层级、中心点、边界 GeoJSON 和区域简介
- 是地图展示和区域聚合的基础表

主要字段：

| 字段名 | 类型 | 含义 |
|---|---|---|
| `region_id` | `BIGINT` | 主键 |
| `parent_region_id` | `BIGINT` | 上级行政区 ID，自关联 |
| `region_name` | `VARCHAR(100)` | 行政区名称 |
| `region_level` | `ENUM` | 行政级别，`province/city/county/township/village` |
| `adcode` | `VARCHAR(20)` | 行政区编码 |
| `center_longitude` | `DECIMAL(10,7)` | 中心经度 |
| `center_latitude` | `DECIMAL(10,7)` | 中心纬度 |
| `boundary_geojson` | `JSON` | 边界 GeoJSON |
| `intro` | `TEXT` | 区域简介 |
| `created_at` | `DATETIME` | 创建时间 |
| `updated_at` | `DATETIME` | 更新时间 |

主外键与索引：

- 主键：`region_id`
- 外键：`parent_region_id -> administrative_region.region_id`
- 唯一：`adcode`
- 索引：`parent_region_id`、`(region_name, region_level)`

设计特点：

- 采用树形结构组织省、市、县、乡镇、村
- `boundary_geojson` 让前端可以直接做区域边界展示

## 4. MySQL 表之间的关系

## 4.1 行政区作为根实体

`administrative_region` 是所有地图型数据的区域根节点。

它和其他表的关系：

- `red_site.region_id -> administrative_region.region_id`
- `hero_person.native_place_region_id -> administrative_region.region_id`
- `historical_event.primary_region_id -> administrative_region.region_id`
- `memorial_hall.region_id -> administrative_region.region_id`
- `red_story.related_region_id -> administrative_region.region_id`

同时它自己还有：

- `parent_region_id -> region_id`

这是一个典型的树形层级结构。

## 4.2 核心实体直接关系

核心实体间通过关系表相连：

- 遗址与事件：`site_event_rel`
- 遗址与人物：`site_hero_rel`
- 事件与人物：`event_hero_rel`
- 纪念馆与遗址：`memorial_site_rel`
- 纪念馆与人物：`memorial_hero_rel`
- 纪念馆与事件：`memorial_event_rel`
- 故事与其他实体：`story_entity_rel`

## 4.3 标签关系

通过 `entity_tag_rel` 把以下实体打标签：

- 遗址
- 人物
- 事件
- 纪念馆
- 故事

## 4.4 来源、媒体、分块关系

补充型关系如下：

- `red_story.source_id -> data_source.source_id`
- `resource_media.source_id -> data_source.source_id`
- `entity_source_rel.source_id -> data_source.source_id`
- `content_chunk.source_id -> data_source.source_id`

这些表不直接改变核心知识图谱结构，但决定了资源证据链和问答质量。

## 5. 当前 MySQL 设计的优点与问题

## 9. 我对当前数据库的判断

如果项目仍然只是“红色文化地图门户”，当前数据库结构是够用的。

但如果项目已经明确转向：

- 河北乡村学校
- 周边本土思政资源
- 课堂活动组织
- 教学问答与讲解

那么数据库**建议尽快升级**，原因有三点：

1. 现在没有 `school` 这个真正的一号业务对象
2. 现在资源类型过于偏红色文化，难以覆盖非遗、公益、养老院、劳动教育等
3. 现在缺少“学校与资源之间的空间关系表”，后续无法稳定做学校周边推荐

## 10. 建议你下一步优先做的数据库调整

建议优先级如下：

1. 新增 `school` 表
2. 新增通用 `local_edu_resource` 表
3. 新增 `school_resource_rel` 表
4. 保留原 `red_site / memorial_hall / historical_event / hero_person / red_story` 作为特殊资源子类型或历史模块
5. 后续再考虑是否把红色资源统一并入 `local_edu_resource`

## 11. 结论

当前数据库设计的特点是：

- 结构完整
- 关系清晰
- 适合红色文化图谱
- 适合做第一阶段地图门户

但它还没有真正切换到“学校中心化”的第二阶段形态。

如果你现在要做样例学校试点，我建议数据库先做“小步升级”，不要一口气推翻：

- 先新增学校表和学校周边资源关系表
- 先让样例学校跑通
- 再决定要不要把原红色文化表进一步抽象重构
