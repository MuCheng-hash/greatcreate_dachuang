# 红韵乡途项目后续进展需求整理

## 1. 项目定位

项目后续应从“红色文化地图门户”升级为“面向乡村学校的本土思政教育资源智能平台”。

平台以乡村学校为中心，采集学校周边可用于思政教育的真实资源，并通过地图、知识图谱和 LLM 能力，帮助教师发现资源、组织教学主题、生成可落地的教学活动方案。

## 2. 核心目标

1. 建立乡村学校基础信息与空间位置数据库。
2. 建立学校周边思政教育资源库。
3. 支持地图展示学校与周边资源关系。
4. 支持基于知识图谱的资源关系查询。
5. 支持基于 RAG 的教学方案生成。
6. 支持教师围绕学校、年级、主题进行智能问答。

## 3. 用户角色

### 3.1 学校教师

教师是核心使用者，主要需求是查找学校周边可用于课堂和实践活动的资源，并生成教学方案。

### 3.2 平台管理员

管理员负责学校、资源、教学方案、数据来源和审核流程管理。

### 3.3 学生

学生主要通过地图和教学活动参与本地文化、红色文化和社会实践学习。

### 3.4 教育管理者

教育管理者关注区域内学校资源覆盖情况、资源质量和思政实践活动开展情况。

## 4. 核心业务流程

```text
学校定位
→ 周边资源采集
→ 资源分类与审核
→ 学校-资源关系计算
→ 知识图谱关联
→ RAG 检索
→ LLM 生成教学方案或问答结果
→ 前端地图展示与教师使用
```

## 5. 功能需求

### 5.1 学校信息管理

系统应支持学校基础信息管理，包括：

- 学校名称
- 学段
- 学校类型
- 所属行政区
- 详细地址
- 经纬度
- 联系人信息
- 审核状态

系统应支持学校注册、审核、修改和停用。

### 5.2 周边思政资源管理

系统应支持多类型本土思政教育资源管理，包括：

- 红色文化资源
- 革命遗址
- 纪念馆
- 英雄人物故事
- 非遗文化资源
- 地方传统文化资源
- 乡贤文化资源
- 爱国主义教育基地
- 劳动教育资源
- 公益实践资源
- 养老院、福利院、社区服务点
- 生态文明教育资源

每个资源应至少包含：

- 资源名称
- 资源类型
- 所属区域
- 地址
- 经纬度
- 简介
- 教育价值
- 适合学段
- 活动建议
- 数据来源
- 审核状态

### 5.3 学校周边资源推荐

系统应根据学校位置计算一定范围内的可用资源，并形成学校与资源之间的关系。

系统应支持：

- 按距离推荐资源
- 按资源类型筛选
- 按适合年级筛选
- 按教学主题筛选
- 展示资源到学校的距离
- 展示推荐理由

### 5.4 地图展示

系统应提供地图页面，展示：

- 当前学校位置
- 学校周边资源点
- 资源分类图标
- 行政区域边界
- 资源详情
- 推荐活动路线
- 学校周边资源统计

地图页面应支持点击学校、资源和区域后加载详情信息。

### 5.5 教学方案生成

系统应支持教师输入：

- 学校
- 年级
- 教学主题
- 活动类型
- 时间长度
- 是否需要实践活动

系统应基于学校周边资源、已有教学方案、知识图谱关系和文本内容检索结果，生成结构化教学方案。

教学方案应包含：

- 教学主题
- 适用年级
- 教学目标
- 资源依据
- 活动流程
- 课前准备
- 现场任务
- 安全提示
- 课后反思
- 评价方式
- 引用来源

### 5.6 智能问答

系统应支持教师围绕学校周边资源进行自然语言提问。

示例问题：

- 某小学附近有哪些适合四年级的红色文化资源？
- 这所学校可以怎样开展敬老志愿服务？
- 某个资源适合设计成什么样的思政课？
- 李大钊相关资源能否和本地学校产生教学关联？

系统回答应包含：

- 自然语言答案
- 相关资源
- 推荐教学主题
- 引用来源
- 后续追问建议

### 5.7 知识图谱关系查询

系统应支持将核心实体同步到 Neo4j，包括：

- 学校
- 行政区
- 红色资源
- 本土思政资源
- 人物
- 事件
- 故事
- 教学主题

系统应支持关系查询，例如：

```text
学校 → 周边资源 → 相关人物 → 相关事件 → 相关故事 → 教学主题
```

前端应至少展示关系链说明，后续可扩展为图谱可视化。

## 6. AI 能力需求

### 6.1 LLM 服务独立部署

LLM 能力应放在独立服务中，不应直接混入业务服务。

LLM 服务主要负责：

- 教学方案生成
- 智能问答
- 资源讲解生成
- 推荐理由生成
- 后续可扩展 Agent 编排

### 6.2 RAG 检索增强生成

系统应实现 RAG 链路：

```text
用户问题
→ 查询学校和资源数据
→ 检索内容分块
→ 查询 Neo4j 图谱关系
→ 组装 Prompt
→ 调用 LLM
→ 返回答案和引用
```

系统应具备：

- 内容分块
- embedding 生成
- 向量库检索
- 图谱事实检索
- Prompt 模板
- 引用来源返回

### 6.3 Agent 能力预留

系统后续应支持轻量 Agent，将复杂问题拆成多个步骤：

```text
识别学校
→ 查询学校坐标
→ 查询周边资源
→ 查询图谱关系
→ 生成教学建议
```

首期不要求复杂 Agent，但接口和服务结构应预留扩展点。

## 7. 数据需求

### 7.1 学校数据

应优先采集河北乡村学校基础数据，包括学校名称、地址、学段、经纬度和所属区域。

### 7.2 资源数据

应围绕学校采集周边资源数据，优先选择可用于教学活动的真实资源。

### 7.3 内容数据

应采集可用于 RAG 的文本内容，包括：

- 资源简介
- 人物故事
- 历史事件
- 地方文化介绍
- 教学活动案例
- 政策或教育材料

### 7.4 数据来源

数据应记录来源，便于审核和回答引用。

## 8. 非功能需求

### 8.1 可演示性

系统必须支持完整演示流程：

```text
学校登录
→ 查看周边资源
→ 点击资源详情
→ 生成教学方案
→ 查看引用来源
```

### 8.2 可扩展性

系统结构应明确分为：

- 业务服务
- LLM 服务
- 数据清洗脚本
- 图谱同步脚本
- 前端页面
- 文档与规格

### 8.3 准确性

LLM 回答必须基于检索结果和引用来源，避免无依据生成。

### 8.4 可维护性

项目目录、服务命名、接口命名和配置命名应保持统一，避免混用 QA、LLM、后端 demo 等临时名称。

## 9. 优先级

### P0：必须完成

- 学校中心化数据结构
- 学校周边资源关系
- 地图展示学校与资源
- LLM 服务真实调用
- 教学方案 RAG 生成
- 基础引用来源展示

### P1：重要功能

- Neo4j 同步与图谱关系查询
- 向量库检索
- 智能问答
- 后台资源审核
- 管理员维护资源和教学方案

### P2：扩展功能

- Agent 多步骤问答
- 语音交互
- 图谱可视化
- 路线规划
- 区域教育资源分析看板

## 10. 验收标准

项目下一阶段至少应满足以下验收标准：

1. 系统中至少有 1 所样例学校和 10 个以上周边资源。
2. 地图能展示学校位置和周边资源。
3. 教师能输入学校、年级、主题并生成教学方案。
4. 生成内容能展示引用来源。
5. 至少有一条 Neo4j 图谱关系查询可演示。
6. LLM 服务与业务服务分离。
7. 前端能完成从学校到资源再到 AI 生成方案的闭环演示。

## 11. 推荐演示场景

建议以一个固定样例学校作为答辩主线：

```text
某乡村小学
→ 四年级
→ 敬老志愿服务 / 红色文化实践 / 本地非遗文化
→ 系统推荐周边资源
→ AI 生成完整教学活动方案
→ 展示引用依据和资源关系
```

该场景能集中体现：

- 学校中心化
- 周边资源可视化
- 知识图谱关联
- RAG 生成
- LLM 辅助教学
- 思政教育落地
