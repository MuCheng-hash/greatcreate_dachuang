# 学校注册与登录模块设计方案

## 1. 文档说明

本文档用于说明如何在当前“乡村学校周边思政教育资源数字地图平台”中新增：

1. 学校注册功能
2. 管理端审核功能
3. 学校账号登录功能
4. 登录后自动定位到账号所属学校并展示周边资源的功能

本方案基于当前项目已有的 `school`、`local_edu_resource`、`school_resource_rel`、`teaching_activity_plan` 等表设计，采用增量扩展方式实现。

## 2. 为什么建议加入该功能

这个功能非常适合当前项目后续发展方向，原因有三点：

1. 项目正在从“资源整理展示”转向“学校中心化平台”，学校账号注册可以让学校成为真正的数据共建主体
2. 学校最了解自身情况和周边真实资源，让学校填写信息能够提高平台数据来源广度
3. 登录后直接定位到该学校并展示周边资源，能够显著提升教师使用效率

因此，这不是一个额外附属功能，而是一个和项目定位高度一致的扩展模块。

## 3. 业务目标

该模块的核心目标是建立一套“学校自注册、平台审核、审核通过后学校账号登录并自动进入本校地图场景”的闭环流程。

业务主线如下：

`学校注册 -> 提交学校信息 -> 管理端审核 -> 通过后生成学校账号绑定 -> 登录 -> 自动进入该学校地图与周边资源页面`

## 4. 当前项目现状判断

结合当前代码结构，可以确认：

1. 当前项目已经有学校主表 `school`
2. 当前项目已经有学校地图详情接口 `/api/school-map/schools/{schoolId}/detail`
3. 当前项目已经有学校后台管理、资源后台管理、学校资源关联管理、活动方案管理等接口
4. 当前项目还没有用户、账号、密码、登录态、权限控制相关表和接口

这意味着：

**学校注册功能可以加，但需要新增完整的账号认证模块，而不是只加一个注册页面。**

## 5. 功能定位

本模块建议分为 4 个子功能：

## 5.1 学校注册

学校端填写：

1. 账号信息
2. 联系人信息
3. 学校基础信息
4. 学校地理位置信息

提交后进入待审核状态。

## 5.2 管理端审核

管理人员在后台查看学校注册申请，对以下内容审核：

1. 学校名称是否真实
2. 学校地址是否合理
3. 经纬度是否准确
4. 账号归属是否可信

审核结果包括：

1. 审核通过
2. 审核驳回

## 5.3 学校账号登录

审核通过后，学校可使用账号密码登录系统。

登录成功后，后端返回：

1. 账号基本信息
2. 绑定学校 ID
3. 学校名称
4. 角色信息

## 5.4 登录后自动进入学校地图

学校账号登录后，系统不必再次依赖浏览器实时定位，而是：

1. 直接读取账号绑定学校的经纬度
2. 自动定位到该学校地图位置
3. 自动加载学校详情
4. 自动展示周边资源与活动方案

## 6. 角色设计

建议至少区分以下角色：

### 6.1 平台管理员

权限：

1. 查看全部注册申请
2. 审核学校注册
3. 启用或停用学校账号
4. 修改学校绑定关系
5. 查看所有学校与资源信息

### 6.2 学校账号

权限：

1. 登录系统
2. 查看本校信息
3. 查看本校周边资源
4. 查看本校活动方案
5. 后续可扩展为补充本校资料、提交本校周边资源建议

## 7. 数据库设计建议

建议新增 2 张核心表。

## 7.1 school_registration 学校注册申请表

用途：

记录学校提交的注册申请，不直接影响正式学校主表。

建议字段：

1. `registration_id` 申请 ID
2. `apply_account` 申请账号
3. `password_hash` 密码哈希
4. `contact_name` 联系人姓名
5. `contact_phone` 联系电话
6. `contact_email` 联系邮箱
7. `school_name` 学校名称
8. `school_alias` 学校别名
9. `school_level` 学段
10. `school_type` 学校类型
11. `school_nature` 办学性质
12. `county_region_id` 县区 ID
13. `township_region_id` 乡镇 ID
14. `address` 地址
15. `longitude` 经度
16. `latitude` 纬度
17. `geo_source_type` 坐标来源
18. `geo_confidence` 坐标置信度
19. `intro` 学校简介
20. `review_status` 审核状态
21. `review_remark` 审核意见
22. `reviewed_by` 审核人
23. `reviewed_at` 审核时间
24. `linked_school_id` 审核通过后绑定的正式学校 ID
25. `created_at`
26. `updated_at`

### 为什么要单独建申请表

原因：

1. 申请信息不应直接写入正式学校主表
2. 审核前可能存在错误信息
3. 需要保留申请历史和审核痕迹

## 7.2 school_user_account 学校账号表

用途：

记录正式可登录的学校账号。

建议字段：

1. `account_id` 账号 ID
2. `username` 登录账号
3. `password_hash` 密码哈希
4. `role_code` 角色编码，建议先用 `school_admin`
5. `school_id` 绑定学校 ID
6. `registration_id` 来源注册申请 ID
7. `display_name` 显示名称
8. `contact_name` 联系人
9. `contact_phone` 联系电话
10. `status` 账号状态
11. `last_login_at` 最近登录时间
12. `created_at`
13. `updated_at`

## 8. 数据表关系

关系建议如下：

1. `school_registration.linked_school_id -> school.school_id`
2. `school_user_account.school_id -> school.school_id`
3. `school_user_account.registration_id -> school_registration.registration_id`

业务关系说明：

1. 注册申请通过后，可新增正式学校账号
2. 正式学校账号必须绑定一所学校
3. 一个学校原则上可先支持一个主账号，后续再扩展多账号

## 9. 审核流程建议

建议采用以下流程：

### 9.1 学校提交注册

系统动作：

1. 校验账号唯一性
2. 密码做哈希处理
3. 保存到 `school_registration`
4. 状态设为 `pending`

### 9.2 管理端审核

如果审核通过：

1. 若学校主表中不存在对应学校，则新建 `school`
2. 若已存在学校，则绑定已有 `school_id`
3. 生成 `school_user_account`
4. 更新申请记录为 `approved`

如果审核驳回：

1. 申请状态设为 `rejected`
2. 写入驳回原因
3. 不生成学校账号

## 10. 登录功能设计

## 10.1 登录方式

当前项目建议先采用最简单稳定的方式：

1. 用户名 + 密码登录
2. 后端校验密码哈希
3. 返回登录成功信息

## 10.2 会话方式建议

如果只是课程项目和当前阶段答辩使用，推荐两种可选方案：

### 方案 A：Session 登录

优点：

1. 实现简单
2. 适合当前 Spring Boot 单体项目
3. 前后端同源静态页接入方便

### 方案 B：JWT 登录

优点：

1. 后续前后端分离更方便
2. 更适合以后扩展移动端或独立前端

### 当前推荐

如果你们现在以“先快速做出可运行成果”为目标，建议先用：

**Session 登录**

因为当前前端就是 Spring Boot 静态页面，这种接入成本最低。

## 11. 登录后自动定位逻辑

这个功能非常容易和现有地图接口衔接。

建议流程：

1. 用户登录成功
2. 后端返回当前账号绑定的 `schoolId`
3. 前端保存当前登录学校信息
4. 首页初始化时优先判断是否已登录学校账号
5. 若已登录，则直接请求：
   `GET /api/school-map/schools/{schoolId}/detail`
6. 根据返回的学校经纬度：
   - 地图中心定位到该学校
   - 打开学校详情抽屉
   - 展示周边资源
   - 展示活动建议

这样就能实现：

**“登录即进入本校地图场景”**

## 12. 接口设计建议

## 12.1 学校注册接口

### 12.1.1 提交注册申请

`POST /api/auth/school-register`

请求内容：

1. 账号
2. 密码
3. 联系人信息
4. 学校基础信息
5. 地址和地理位置

返回：

1. 注册申请 ID
2. 当前状态

## 12.2 登录接口

### 12.2.1 学校账号登录

`POST /api/auth/login`

请求内容：

1. username
2. password

返回内容建议：

1. accountId
2. username
3. roleCode
4. schoolId
5. schoolName

## 12.3 当前登录信息接口

### 12.3.1 获取当前用户信息

`GET /api/auth/me`

返回内容：

1. 登录账号信息
2. 绑定学校信息

## 12.4 审核接口

### 12.4.1 注册申请列表

`GET /api/admin/registrations`

### 12.4.2 注册申请详情

`GET /api/admin/registrations/{registrationId}`

### 12.4.3 审核通过

`POST /api/admin/registrations/{registrationId}/approve`

### 12.4.4 审核驳回

`POST /api/admin/registrations/{registrationId}/reject`

## 13. 前端页面建议

建议新增以下页面：

### 13.1 学校注册页

功能：

1. 填写账号密码
2. 填写学校基本信息
3. 填写学校地理位置
4. 提交注册申请

### 13.2 学校登录页

功能：

1. 输入账号密码
2. 登录成功后跳转首页

### 13.3 注册审核页

放在当前后台管理台中。

功能：

1. 查看注册申请列表
2. 查看申请详情
3. 审核通过或驳回

## 14. 风险与注意点

## 14.1 不建议学校直接改正式学校主表

原因：

1. 可能出现错误位置、重复学校、虚假账号
2. 审核前不应污染正式数据

## 14.2 密码不能明文存储

必须：

1. 使用密码哈希
2. 不要把密码写到普通业务表中

## 14.3 学校名称可能重复

建议：

1. 审核时优先匹配已有学校
2. 不要仅凭学校名称自动入库

## 14.4 登录后自动定位应优先使用学校绑定坐标

不要仍然只依赖浏览器定位，否则学校账号登录场景意义会减弱。

## 15. 实施优先级建议

建议按以下顺序推进：

1. 先新增注册申请表和学校账号表
2. 先做注册申请提交接口
3. 再做管理端审核接口
4. 再做登录接口和当前用户信息接口
5. 最后改首页逻辑，实现登录后自动进入本校地图场景

## 16. 结论

这个功能完全可以加，而且很适合你们项目。

从业务价值看，它能让学校从“被展示对象”变成“平台共建主体”；从技术实现看，它也能和当前已经完成的学校、资源、周边关联、活动方案模块自然衔接。

如果后续真的要落地，我建议不要只做“注册页面”，而是要把它做成：

**注册申请 + 管理审核 + 学校账号登录 + 登录后自动进入本校地图**

这样这个功能才是完整、合理、能演示、也能答辩的。
