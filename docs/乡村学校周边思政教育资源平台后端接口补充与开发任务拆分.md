# 乡村学校周边思政教育资源平台后端接口补充与开发任务拆分

## 1. 文档说明

本文档用于在现有项目基础上，明确：

1. 当前后端已完成内容
2. 当前后端缺失接口清单
3. 推荐的数据传输对象与控制器划分
4. 开发任务拆分与优先级

本文档面向后续开发实施，尽量与当前 `business-service` 工程结构保持一致。

## 2. 当前后端现状判断

结合当前代码，项目已经具备以下基础能力：

### 2.1 已完成部分

1. 已有学校、资源、学校资源关系、活动方案等核心实体类
2. 已有对应 Service、Mapper 基础结构
3. 已有学校地图展示控制器 `SchoolMapController`
4. 已有学校列表、附近学校、学校详情聚合接口
5. 已有学校样例 SQL 和学校模块数据库方案

### 2.2 当前缺失部分

当前最明显的缺口不是数据库，而是“后台管理能力”还不完整，主要缺：

1. 学校新增、编辑、审核接口
2. 资源新增、编辑、审核接口
3. 学校与资源关系维护接口
4. 教学活动方案维护接口
5. Excel 导入接口
6. 统计分析接口
7. 主题化资源查询接口

### 2.3 现阶段判断

也就是说，你们当前项目已经有“展示端雏形”，但还没有真正形成“可持续录入和管理数据的平台后台”。

所以接下来开发重点应从“只读展示接口”转向“增删改查 + 审核 + 导入 + 统计”。

`/api/admin/resources`

## 4.3 SchoolResourceRelAdminController

职责：

建议路径前缀：

`/api/admin/school-resource-rel`

## 4.4 TeachingActivityPlanAdminController

职责：

负责教学活动方案维护。

## 5. 缺失接口清单

以下接口建议优先补齐。

## 5.1 学校管理接口

### 5.1.1 新增学校

`POST /api/admin/schools`

用途：

新增学校基础信息。

建议请求字段：

1. `schoolCode`
2. `schoolName`
3. `schoolAlias`
4. `regionId`
5. `countyRegionId`
6. `townshipRegionId`
7. `villageRegionId`
8. `schoolLevel`
9. `schoolType`
10. `schoolNature`
11. `ruralSchool`
12. `teachingPoint`
13. `address`
14. `longitude`
15. `latitude`
16. `geoSourceType`
17. `geoConfidence`
18. `intro`

### 5.1.2 编辑学校

`PUT /api/admin/schools/{schoolId}`

用途：

更新学校基础信息与定位信息。

### 5.1.3 查询学校详情

`GET /api/admin/schools/{schoolId}`

用途：

后台编辑页数据回显。

### 5.1.4 学校分页列表

`GET /api/admin/schools`

建议支持参数：

1. `keyword`
2. `countyRegionId`
3. `townshipRegionId`
4. `schoolLevel`
5. `reviewStatus`
6. `pageNum`
7. `pageSize`

### 5.1.5 提交审核

`POST /api/admin/schools/{schoolId}/submit-review`

用途：

将学校状态从 `draft` 调整为 `pending`。

### 5.1.6 审核通过

`POST /api/admin/schools/{schoolId}/approve`

用途：

将学校状态调整为 `approved`。

### 5.1.7 审核驳回

`POST /api/admin/schools/{schoolId}/reject`

用途：

将学校状态调整为 `rejected` 并记录意见。

### 5.1.8 导入学校

`POST /api/admin/schools/import`

用途：

通过 Excel 批量导入学校。

## 8. 推荐开发顺序

为了尽快做出成果，建议按以下顺序补后端。

## 8.1 第一阶段：补齐核心 CRUD

优先级最高。

任务：

1. 学校管理接口
2. 资源管理接口
3. 学校资源关系管理接口
4. 活动方案管理接口

完成标准：

1. 可以在后台手工录入学校
2. 可以录入资源
3. 可以为学校绑定资源
4. 可以录入活动建议

## 8.2 第二阶段：补审核流

任务：

1. 学校审核接口
2. 资源审核接口
3. 关系审核状态处理
4. 审核意见记录

完成标准：

1. 草稿、待审核、已通过、已驳回状态可流转

## 8.3 第三阶段：补导入能力

任务：

1. 学校 Excel 导入
2. 资源 Excel 导入
3. 导入错误提示
4. 导入结果统计

完成标准：

1. 样例学校和样例资源可通过模板批量导入

## 8.4 第四阶段：补统计分析

任务：

1. 学校数统计
2. 资源数统计
3. 资源分类统计
4. 学校覆盖情况统计

完成标准：

1. 可以形成后台数据总览页面

## 9. 详细任务拆分建议

以下拆分方式适合 2 到 4 人小团队并行推进。

## 9.1 任务包 A：数据库与样例数据

负责人工作：

1. 执行学校模块 SQL
2. 导入样例学校数据
3. 导入样例资源数据
4. 验证关联关系和活动方案样例

交付物：

1. 本地可运行数据库
2. 可查询样例数据

## 9.2 任务包 B：学校与资源后台接口

负责人工作：

1. 新建学校管理控制器
2. 新建资源管理控制器
3. 编写创建、编辑、详情、列表接口
4. 补请求对象与分页返回结构

交付物：

1. 学校后台 CRUD 接口
2. 资源后台 CRUD 接口

## 9.3 任务包 C：关系与活动接口

负责人工作：

1. 新建学校资源关系管理控制器
2. 新建活动方案管理控制器
3. 编写关系维护和活动维护接口
4. 联调学校详情聚合逻辑

交付物：

1. 学校资源关系管理接口
2. 活动方案管理接口

## 9.4 任务包 D：导入与统计

负责人工作：

1. 设计 Excel 导入模板

2. 资源导入接口

如果时间紧，后端最少要保证以下内容可用：

1. 学校后台 CRUD
2. 资源后台 CRUD
3. 学校与资源关系 CRUD

## 12. 结论

当前项目后端最需要补的，不是继续增加概念设计，而是尽快补齐“管理型接口”。

只要把学校、资源、关系、活动方案这四组后台接口补全，你们的平台就会从“演示型地图原型”升级为“可持续建设的数据平台”。后续不管是做前端后台页面、答辩演示，还是继续扩展智能功能，都会顺很多。
