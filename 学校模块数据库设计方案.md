# 学校模块数据库设计方案

## 1. 设计目标

当前项目数据库是以“行政区 + 红色文化资源”为主轴设计的，适合第一阶段的红色文化地图门户。

但项目下一阶段已经明确转向：

- 河北乡村学校
- 学校周边本土思政教育资源
- 教学活动组织
- 问答讲解与实践建议

因此数据库需要从“区域中心”逐步升级为“学校中心”。

本方案的设计目标是：

1. 不推翻现有红色文化库
2. 在现有 MySQL 上平滑增加学校模块
3. 让样例学校试点能尽快落地
4. 为后续全河北扩展预留空间

## 2. 设计原则

### 2.1 保留现有红色文化主表

以下表建议继续保留，不要推翻：

- `administrative_region`
- `red_site`
- `hero_person`
- `historical_event`
- `memorial_hall`
- `red_story`
- `tag_info`
- `entity_tag_rel`
- `resource_media`
- `entity_source_rel`
- `content_chunk`
- `audit_log`

原因：

- 这些表已经和前后端、地图聚合、Neo4j 同步逻辑绑定
- 直接删除或重做会导致项目现有功能全部返工

### 2.2 新增学校主轴

新增模块后，系统的第一业务对象应该变成：

- `school`

后续地图展示和接口设计都应优先围绕学校，而不是只围绕乡镇。

### 2.3 资源做“通用化”而不是只做“红色化”

如果继续只用 `red_site`、`memorial_hall` 这套命名，会限制后面扩展。

因此建议新增：

- `local_edu_resource`

把本土思政资源统一管理，再让红色资源作为其中一种资源类别。

## 3. 建议新增的核心表

## 3.1 `school` 学校表

用途：

- 作为平台一号对象
- 存学校基础信息、空间坐标、乡村属性、审核状态

核心字段建议：

- 基础标识：`school_id`、`school_code`、`school_name`
- 行政区：`region_id`、`county_region_id`、`township_region_id`、`village_region_id`
- 学校属性：`school_level`、`school_type`、`school_nature`
- 乡村属性：`is_rural_school`、`is_teaching_point`
- 定位信息：`longitude`、`latitude`、`poi_name`、`poi_type`、`geo_confidence`
- 审核与状态：`review_status`、`is_active`

设计考虑：

- `region_id` 作为统一所属行政区字段
- `county_region_id / township_region_id / village_region_id` 让后续学校统计更方便
- `geo_confidence` 便于人工核验地图坐标质量

## 3.2 `school_geo_record` 学校坐标记录表

用途：

- 记录学校坐标来源与修订过程
- 支持一个学校多次定位、多次人工复核

为什么建议单独建表：

- 学校定位很容易出现“POI 匹配不准、同名学校冲突、人工修正”问题
- 如果只有 `school` 主表，后面很难追踪坐标是怎么改过来的

适合记录的信息：

- 坐标值
- 来源方式
- POI 名称、POI 地址、POI 类型
- 置信度
- 是否人工审核
- 审核结果
- 当前是否生效

## 3.3 `local_edu_resource` 本土思政教育资源表

用途：

- 统一存储学校周边可用于思政教育的本土资源

资源大类建议统一为：

- `red_culture` 红色文化
- `intangible_culture` 非遗传统文化
- `traditional_culture` 中华优秀传统文化
- `local_history` 地方历史文化
- `public_culture` 公共文化资源
- `labor_education` 劳动教育资源
- `public_welfare` 公益实践资源
- `ecological_civilization` 生态文明教育资源
- `patriotism_base` 爱国主义教育基地
- `social_practice` 社会实践资源
- `other`

设计考虑：

- 这样后面不只是纪念馆、遗址，像养老院、村史馆、非遗工坊、文化礼堂也能统一入库
- 后续地图和问答系统都能按统一资源模型工作

## 3.4 `school_resource_rel` 学校-资源关系表

用途：

- 表达“某所学校周边有哪些资源”
- 这是学校中心化平台最关键的一张关系表

为什么不能直接只靠空间计算：

- 地图距离只是一个维度
- 真正适合课堂使用，还要考虑交通方式、可达性、教育主题、合作关系

因此建议存这些信息：

- 学校 ID
- 资源 ID
- 距离米数
- 推荐出行方式
- 预计时长
- 可达性等级
- 优先级
- 教育主题摘要

## 3.5 `teaching_activity_plan` 教学活动建议表

用途：

- 存“学校如何使用资源做课堂/实践活动”的结果

适合存的内容：

- 主题
- 活动类型
- 适合年级
- 教学目标
- 活动流程
- 前期准备
- 安全提示
- 预期成果

意义：

- 这张表能把平台从“资源地图”升级为“教学支持平台”

## 4. 与现有旧表的关系

## 4.1 与 `administrative_region` 的关系

新增学校和资源都应继续挂接到行政区表：

- `school.region_id -> administrative_region.region_id`
- `local_edu_resource.region_id -> administrative_region.region_id`

这样可以继续复用你们现有的：

- 区域树
- 地图中心点
- 边界 GeoJSON
- 省市县乡镇聚合逻辑

## 4.2 与 `data_source` 的关系

新增表建议统一保留 `source_id`：

- `school.source_id`
- `local_edu_resource.source_id`
- `school_resource_rel.source_id`
- `teaching_activity_plan.source_id`

这样可以继续复用原有来源治理体系。

## 4.3 与通用多态表的关系

建议把这些旧表的 `entity_type` 扩展到支持新实体：

- `entity_tag_rel`
- `resource_media`
- `entity_source_rel`
- `content_chunk`
- `audit_log`
- `story_entity_rel`

新增实体类型建议包括：

- `school`
- `resource`
- `activity_plan`

这样做的好处：

- 不用再重复造一套学校标签表、学校媒体表、学校来源表
- 能最大程度复用现有数据库框架

## 5. 新模块与旧红色文化表怎么衔接

这是你们设计里最关键的问题之一。

我建议分两步走。

### 第一步：并行存在

先保留：

- `red_site`
- `historical_event`
- `hero_person`
- `memorial_hall`
- `red_story`

同时新增：

- `school`
- `local_edu_resource`
- `school_resource_rel`

这样可以用最小改动做出样例学校闭环。

### 第二步：逐步抽象

等学校试点稳定后，再考虑：

- 是否把 `red_site` / `memorial_hall` 中能用于学校周边教育的内容同步映射到 `local_edu_resource`
- 是否把部分故事和活动建议统一进教学资源体系

也就是说：

- 短期：双轨并行
- 中期：资源统一抽象

## 6. 为什么不建议现在就彻底重构

虽然从长期看，做一个更纯粹的“统一教育资源模型”会更优雅，但现阶段不建议直接推翻重来，原因是：

1. 当前已有地图接口和聚合逻辑都依赖旧表
2. 当前 Neo4j 同步设计已经围绕旧实体建立
3. 你们需要先尽快做出样例学校成果
4. 全量重构会拉长周期，影响答辩和阶段产出

所以更合理的是：

- 先补学校模块
- 再慢慢迁移业务重心

## 7. 具体落地建议

建议分三步实施：

### 第一步：只建 4 张表

- `school`
- `school_geo_record`
- `local_edu_resource`
- `school_resource_rel`

这是最小闭环。

### 第二步：做学校样例试点

先挑 3 到 5 所学校，录入：

- 学校基本信息
- 坐标信息
- 周边资源
- 学校与资源关系

### 第三步：补教学活动表

在样例资源稳定后，再建：

- `teaching_activity_plan`

这样能把“资源展示”进一步升级到“教学落地”。

## 9. 我的最终建议

如果你现在要我帮你判断数据库怎么走，我的建议是：

1. 不推翻现有红色文化库
2. 先新增学校模块
3. 用样例学校把学校中心化流程跑通
4. 以后再决定是否做统一资源重构

一句话总结：

**当前最优方案不是“重做数据库”，而是“在现有数据库上加一个学校主轴”。**
